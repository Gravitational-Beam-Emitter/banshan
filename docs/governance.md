# 半山项目治理方案

## 一、核心原则

1. **单任务原则**：一次只做一个最小的可验证功能模块，完成后停下来汇报，等待确认再继续。
2. **自检原则**：每次输出代码后，自动运行静态检查（如果有测试命令则运行），并报告结果。出现错误必须自动修复最多3次，仍失败则停止并说明原因。
3. **可追溯原则**：所有代码文件头部标注版本号和最后修改简述。每次任务完成后生成一个简短的 `CHANGELOG.md` 条目。
4. **人机边界原则**：不得擅自引入未经用户同意的第三方服务；不得在代码中埋点或收集用户信息；不得做超出当前任务范围的优化。

## 二、任务拆解规范

当用户给出功能需求时，按以下步骤输出任务拆解清单：

1. **分解为子任务**：每个子任务不超过 200 行代码改动，且可独立测试
2. **标注依赖**：哪些子任务需要先完成其他子任务
3. **预估改动范围**：涉及哪些文件（新建/修改）
4. **请求确认**：输出拆解清单后，等用户回复"执行"才开始写代码

示例格式：
```
## 任务拆解：实现症状输入与分析
1. 创建 SymptomInput.vue（新建，约40行）- 依赖：无
2. 在 api/analyze.js 中实现 DeepSeek 调用函数（新建，约60行）- 依赖：环境变量配置
3. 在 Home.vue 中集成输入框和调用逻辑（修改，约30行）- 依赖：1, 2
4. 创建 ResultCard.vue 展示等级+解释+行动（新建，约50行）- 依赖：3
请确认是否按此顺序执行。
```

## 三、代码生成与文件管理

### 3.1 文件头模板
```javascript
/**
 * 半山项目 - [文件名]
 * 功能简述：[...]
 * 版本: 0.1.0
 * 最后修改: YYYY-MM-DD
 * 修改说明: 初始创建
 */
```

### 3.2 目录结构
```
banshan/
├── frontend/
│   ├── src/
│   │   ├── components/
│   │   ├── pages/
│   │   ├── utils/
│   │   └── App.vue
│   └── package.json
├── backend/
│   └── api/
├── shared/
│   └── prompts.js
├── .env.example
├── README.md
└── CHANGELOG.md
```
不得随意新增顶层目录，除非先说明并获得同意。

### 3.3 代码风格
- 使用 ES6+ 语法
- 函数必须有注释说明参数和返回值
- 避免超过 80 字符的行
- 所有用户可见的文本字符串放 `shared/strings.js`

## 四、测试与验收流程

### 4.1 子任务验证
每次完成一个子任务后必须：
1. 运行该功能的单元测试或描述手动验证步骤
2. 输出验证结果：`✅ 通过` 或 `❌ 失败 – 错误信息…`
3. 如果失败，自动修复（最多3次），修复后重新验证
4. 3次仍失败，停止并报告："无法自动修复，需要人工介入。错误详情：..."

### 4.2 集成测试
每完成一个完整功能闭环后：
- 启动开发服务器或提供预览链接
- 给出手动测试路径
- 等用户确认"通过"或"不通过"

## 五、版本与迭代管理

### 5.1 版本号规则
- `0.x.x`：MVP 开发阶段
- `1.0.0`：第一个可公开发布的版本
- 每次功能里程碑完成，自动更新版本号并追加 CHANGELOG

### 5.2 变更记录格式
```markdown
## [0.2.0] - 2026-06-20
### 新增
- 症状输入页面
- AI 分析结果卡片
### 修改
- 调整 API 调用超时时间为 10 秒
### 修复
- 修复历史记录页面删除按钮失效
```

## 六、角色与沟通格式

### 6.1 角色定义
- **PO（用户）**：提用户故事、验收功能、提供外部资源
- **Claude Code**：拆解任务、生成代码、自测修复、汇报进度

### 6.2 沟通格式
每个回复必须以以下结构开头：
```
[任务进度] 当前阶段：xxx
[状态] 进行中 / 等待确认 / 阻塞
[问题]（如果有）...
[下一步] 我需要你...
```

## 七、禁止事项（MVP 阶段）
1. 不引入数据库（用 localStorage 或前端内存）
2. 不做用户系统（登录、注册、密码找回）
3. 不做付费集成
4. 不做复杂状态管理（超过5个页面且共享状态复杂时再考虑）
5. 不做 SSR、PWA、离线支持
